# Пользовательская инструкция Agile Manager

## 1. Назначение приложения

Agile Manager - учебное веб-приложение для организации командной работы по Agile. В нем есть Kanban-доска, спринты, задачи, отметки выполнения, аналитика, журнал активности, управление пользователями и realtime-обновления между открытыми вкладками.

Приложение разделено на две страницы:

- `/` - вход в систему.
- `/app.html` - основное рабочее пространство.

Без авторизации рабочее пространство не открывается.

## 2. Вход в систему

На странице входа нужно указать логин и пароль.

Демо-учетные записи:

- Виктор Пелевин, руководитель: `manager` / `manager2026`
- Мишель Фуко, работник: `worker1` / `user2026`
- Роберт Смит, работник: `worker2` / `user2026`

После входа справа вверху отображается текущая учетная запись. Для завершения работы используется кнопка `Выйти`.

Если несколько пользователей работают одновременно, открытые страницы автоматически получают обновления. Например, если руководитель переносит карточку в другую колонку, у работника доска обновляется почти сразу.

## 3. Роли

### Руководитель

Руководитель управляет рабочим процессом команды.

Доступные действия:

- создавать и редактировать задачи;
- назначать исполнителей;
- переносить задачи между колонками;
- видеть очередь `Проверка`;
- создавать и редактировать спринты;
- открывать карточку спринта со списком задач;
- создавать и редактировать пользователей;
- смотреть аналитику и журнал активности.

### Пользователь

Пользователь работает со своими задачами.

Доступные действия:

- смотреть доску;
- использовать фильтр `Мои`;
- открывать задачу двойным кликом;
- читать название и описание задачи;
- отмечать свою задачу как выполненную на текущем этапе;
- при необходимости оставлять комментарий к выполнению.

Пользователь не может создавать задачи, редактировать задачи, переносить карточки, менять спринты или управлять пользователями.

## 4. Доска задач

Доска состоит из колонок:

- `Backlog`
- `To Do`
- `In Progress`
- `Review`
- `Done`

На панели над доской есть поиск, фильтр по исполнителю, фильтр по спринту и переключатель `Мои`.

Фильтр `Мои` показывает только задачи, назначенные на текущего пользователя. Это удобно для работников, которым не нужно каждый раз искать свои карточки среди всех задач команды.

Длинное описание на карточке ограничено по высоте. Полный текст открывается двойным кликом по задаче.

## 5. Создание и редактирование задач

Создание и редактирование задач доступно руководителю.

Чтобы создать задачу:

1. Нажать `+` над доской.
2. Заполнить название, описание и параметры задачи.
3. Выбрать исполнителя и спринт.
4. Нажать `Сохранить`.

Чтобы отредактировать задачу:

1. Дважды кликнуть по карточке.
2. Изменить нужные поля.
3. Нажать `Сохранить`.

Обычный пользователь при двойном клике видит только содержание задачи: название и описание.

## 6. Отметка выполнения

Работник нажимает `Выполнено` на своей задаче, когда завершил работу на текущем этапе.

Комментарий необязателен:

- если есть важные детали, их можно указать;
- если деталей нет, поле можно оставить пустым.

Отметка выполнения не переносит задачу в другую колонку. Она отправляет задачу руководителю в очередь `Проверка`.

## 7. Очередь проверки

Вкладка `Проверка` доступна руководителю.

В ней отображаются задачи, которые работники отметили как выполненные, но которые еще не были перенесены дальше по процессу.

В карточке проверки руководитель видит:

- название задачи;
- описание;
- исполнителя;
- комментарий исполнителя, если он был оставлен;
- текущий статус.

В этой вкладке нельзя редактировать задачу. Руководитель только принимает результат и переносит задачу на следующий этап одной кнопкой.

После переноса задачи на следующий этап отметка выполнения сбрасывается.

## 8. Спринты

Раздел `Спринты` показывает список спринтов и прогресс по ним.

В карточке спринта отображается:

- название;
- цель;
- статус;
- период;
- количество завершенных задач;
- прогресс-бар;
- ретроспектива, если она заполнена.

Кнопка `Открыть` показывает отдельное окно спринта со списком задач внутри. Так проще понять, какие задачи входят в конкретный спринт и кто за них отвечает.

Руководитель может создавать и редактировать спринты. Пользователь может только просматривать информацию.

## 9. Пользователи

Вкладка `Пользователи` доступна руководителю.

Возможности:

- добавить нового пользователя;
- изменить логин, имя, email, роль и пароль;
- оставить пароль пустым при редактировании, чтобы он не менялся.

Доступные роли:

- `Администратор`
- `Руководитель`
- `Пользователь`

## 10. Аналитика

Раздел `Аналитика` показывает:

- активные задачи;
- завершенные задачи;
- velocity;
- WIP;
- загрузку участников команды;
- ближайшие дедлайны.

Эти данные помогают руководителю видеть нагрузку и планировать работу.

## 11. Журнал активности

Раздел `Активность` хранит события проекта: создание задач, изменение статусов, отметки выполнения, обновление спринтов и пользователей.

Toast-уведомления появляются временно и исчезают через несколько секунд. Журнал активности остается доступен отдельной вкладкой.

Уведомления создаются для всех пользователей отдельно. Если один пользователь закрывает уведомление, у остальных оно остается. Новые уведомления приходят в открытые вкладки автоматически.

## 12. Типовой сценарий

1. Руководитель входит в систему.
2. Создает пользователей, если нужны новые участники.
3. Создает спринт.
4. Создает задачи и назначает исполнителей.
5. Пользователь входит в систему и включает фильтр `Мои`.
6. Пользователь выполняет задачу и нажимает `Выполнено`.
7. Руководитель открывает вкладку `Проверка`.
8. Руководитель переносит задачу на следующий этап.
9. Команда видит изменения на доске без ручного обновления страницы.
10. Команда отслеживает прогресс через спринты, аналитику и журнал активности.

## 13. Ограничения текущей версии

Текущая версия подходит для учебной демонстрации и локального запуска.

Приложение поддерживает два режима запуска: один процесс или набор микросервисов. Для пользователя интерфейс не меняется: вход, доска, проверка, уведомления и realtime-обновления работают одинаково.

Для запуска микросервисов через Docker используется команда `docker compose up --build`. После старта приложение открывается по адресу `http://localhost:8080`.

Ограничения:

- микросервисы используют одну PostgreSQL-базу;
- данные хранятся в PostgreSQL, но без отдельной миграционной системы;
- пароли демонстрационные;
- нет удаления пользователей;
- нет отдельной истории изменений внутри карточки задачи;
- нет реального многопользовательского разграничения на уровне базы данных.
